# Спецификация: CLI-инструмент для изучения иностранных слов (Telegram-бот)

## 1. Цель
Создать CLI-инструмент для отдельного Telegram-бота, который используется для интервального изучения слов/фраз с карточками (front/back).

## 2. Контекст использования
- Пользователь взаимодействует с инструментом только через Telegram-бота.
- CLI используется как служебный слой для:
  - запуска бота,
  - администрирования колод,
  - импорта/экспорта карточек,
  - просмотра статистики.

## 3. Функциональные требования

### 3.1 Поддержка нескольких языков
- Система должна поддерживать изучение карточек на нескольких языках.
- В одной колоде допускаются карточки только одной языковой пары (например, `EN -> RU`), либо явно заданной конфигурации колоды.
- Пользователь может иметь несколько колод для разных языков.

### 3.2 Карточка
Каждая карточка должна содержать минимум:
- `id`
- `front` — лицевая сторона (слово/фраза)
- `back` — оборотная сторона (перевод/пояснение/пример)
- `language_from`
- `language_to`
- `status` (`active`, `snoozed`, `removed`)
- `snoozed_until` (timestamp, если применимо)
- `created_at`, `updated_at`

### 3.3 Отображение карточки в Telegram
- В сообщении показывается лицевая сторона (`front`).
- Оборотная сторона (`back`) выводится под спойлером.
- Формат спойлера: Telegram MarkdownV2 `||...||` или эквивалент через HTML, совместимый с Telegram Bot API.

### 3.4 Кнопки и логика поведения
Для каждой показанной карточки доступны 3 кнопки:

1. `❌ Не помню`
- Карточка остается в колоде.
- Карточка продолжает участвовать в выборе следующей карточки.
- Никакая пауза (snooze) не ставится.

2. `✅ Помню`
- Карточка исключается из показа на следующие 24 часа.
- Реализация: `status = snoozed`, `snoozed_until = now + 24h`.

3. `🗑 Убрать`
- Карточка удаляется из активной колоды.
- Рекомендуемая реализация: мягкое удаление (`status = removed`) с возможностью восстановления.

## 4. Правила выбора следующей карточки
- В выборку попадают только карточки со `status = active`.
- Карточки со `status = snoozed` попадают в выборку только если `now >= snoozed_until`.
- Карточки со `status = removed` в выборке не участвуют.
- При отсутствии доступных карточек бот сообщает пользователю, что новых карточек сейчас нет.

## 5. CLI-команды (MVP)

### 5.1 Управление ботом
- `bot start` — запуск Telegram-бота.
- `bot stop` — остановка процесса бота (опционально, если управление внешним процесс-менеджером).

### 5.2 Управление колодами
- `deck create --name <name> --from <lang> --to <lang>`
- `deck list`
- `deck delete --id <deck_id>`

### 5.3 Управление карточками
- `card add --deck <deck_id> --front "..." --back "..."`
- `card import --deck <deck_id> --file <csv|json>`
- `card list --deck <deck_id> [--status active|snoozed|removed]`
- `card remove --id <card_id>`
- `card restore --id <card_id>`

### 5.4 Сессии/практика
- `practice --deck <deck_id>` — инициировать режим показа карточек (если требуется ручной старт, иначе только через бота).

## 6. Нефункциональные требования
- Простота (KISS): минимальная архитектурная сложность для MVP.
- Надежность хранения: данные карточек не должны теряться при перезапуске.
- Идемпотентная обработка callback-кнопок (защита от двойных нажатий/повторов).
- Логи действий пользователя (минимум: выбор кнопки и card_id).

## 7. Модель данных (минимум)

### Таблица `decks`
- `id` (PK)
- `name`
- `language_from`
- `language_to`
- `created_at`
- `updated_at`

### Таблица `cards`
- `id` (PK)
- `deck_id` (FK -> decks.id)
- `front`
- `back`
- `status` (`active`, `snoozed`, `removed`)
- `snoozed_until` (nullable)
- `created_at`
- `updated_at`

### Таблица `user_actions` (опционально для аналитики)
- `id` (PK)
- `telegram_user_id`
- `card_id`
- `action` (`dont_remember`, `remember`, `remove`)
- `created_at`

## 8. Telegram UX (MVP)
1. Бот отправляет сообщение с `front`.
2. Ниже в том же сообщении показывает `back` под спойлером.
3. Под сообщением inline-кнопки:
   - `❌ Не помню`
   - `✅ Помню`
   - `🗑 Убрать`
4. После нажатия кнопки:
   - фиксируется действие,
   - состояние карточки обновляется,
   - отправляется следующая карточка (если есть).

## 9. Варианты реализации

### Вариант A (рекомендуется для быстрого MVP)
**Стек:** Python + aiogram + SQLite + Typer/Click

**Плюсы:**
- Быстрый старт и короткое время до результата.
- Отличная экосистема для Telegram-ботов.
- SQLite достаточно для одного пользователя/малой команды.

**Минусы:**
- Ограниченная масштабируемость при росте нагрузки.

**Когда выбирать:**
- Нужен рабочий прототип максимально быстро.

### Вариант B (JS/TS экосистема)
**Стек:** Node.js (TypeScript) + Telegraf + Prisma + SQLite/PostgreSQL

**Плюсы:**
- Единый стек JS/TS.
- Удобная типизация доменной модели через Prisma.
- Простой путь к переходу на PostgreSQL.

**Минусы:**
- Чуть выше сложность инфраструктуры, чем в варианте A.

**Когда выбирать:**
- Команда уже активно использует TypeScript.

### Вариант C (производительность и один бинарник)
**Стек:** Go + go-telegram-bot-api + Cobra + SQLite/PostgreSQL

**Плюсы:**
- Один статический бинарник, удобно деплоить.
- Хорошая производительность и надежность.

**Минусы:**
- Более высокий порог разработки для быстрого MVP.

**Когда выбирать:**
- Нужен production-first подход и минимальная зависимость от рантайма.

## 10. Рекомендация по запуску
Для текущих требований и принципа KISS:
1. Начать с **Варианта A** (Python + aiogram + SQLite).
2. Зафиксировать контракт callback-действий и модель данных.
3. После валидации UX при необходимости перейти на PostgreSQL без изменения бизнес-логики.

## 11. Критерии приемки MVP
- Бот показывает карточку с `front` и `back` под спойлером.
- Кнопка `Не помню` оставляет карточку в ротации.
- Кнопка `Помню` скрывает карточку на 24 часа.
- Кнопка `Убрать` исключает карточку из колоды.
- Поддерживается не менее 2 языковых колод у одного пользователя.
- Базовые CLI-команды для добавления/импорта карточек работают.
